## What This Is

A React survey app with real-time voice transcription. Users answer questions by typing or speaking ‚Äî voice input streams through Amazon Transcribe via a browser WebSocket so words appear live as they talk. Surveys are data-driven: questions and branching logic live in DynamoDB, not in code.

## Hard Rules

These are non-negotiable. Violating any of them will break the app or the security model.

1. **No Web Speech API** ‚Äî fails silently on iOS. All transcription goes through Amazon Transcribe Streaming.
2. **No direct AWS calls from the frontend** except the Transcribe WebSocket (needs browser-to-AWS for real-time). Credentials come from a Lambda via GraphQL.
3. **No hardcoded questions** ‚Äî questions are fetched from DynamoDB. The frontend is generic.
4. **No API keys or secrets in code** ‚Äî IAM roles handle everything. Transcribe creds are temporary and scoped.
5. **No S3 for transcription** ‚Äî audio streams directly from browser to Transcribe. Never stored.
6. **Audio sample rates must match** ‚Äî `getUserMedia` (16000) must equal `MediaSampleRateHertz` (16000). Mismatches = garbled audio.
7. **HTTPS required** ‚Äî `getUserMedia` needs a secure context. Localhost is exempt in dev.
8. **Amplify Gen 2 only** ‚Äî no Gen 1 CLI commands (`amplify add`, `amplify push`). Backend is TypeScript in `amplify/`.
9. **Don't edit `amplify_outputs.json`** ‚Äî it's auto-generated by sandbox/deploy.

## Architecture

### Security Model
- No API keys in the codebase. AWS access = IAM roles provisioned by Amplify.
- Frontend talks to backend via AppSync (data) and a credential-generating Lambda (transcription).
- Lambda functions get only the IAM permissions granted in `backend.ts` ‚Äî least-privilege.
- DynamoDB auth rules are per-model. Guests can read questions and create responses.
- Transcribe credentials are short-lived (15 min), scoped to `transcribe:StartStreamTranscription` only.

### Voice Transcription Flow
1. User taps "Record."
2. Frontend calls Lambda (via GraphQL mutation) ‚Üí gets temporary Transcribe credentials.
3. Frontend opens WebSocket to Amazon Transcribe Streaming using `@aws-sdk/client-transcribe-streaming`.
4. Browser captures mic audio via `getUserMedia`, PCM-encodes it, streams chunks over WebSocket.
5. Transcribe returns partial results live ‚Äî words appear on screen progressively.
6. Partials (may change) show in italic. Finals (confirmed) are appended to stable transcript.
7. User sees transcript in real time. Wrong? Tap "Re-record." Correct? Confirm.
8. On confirm, final text is saved to the response field. No audio stored.

**Why streaming instead of batch:** No S3, no polling, no wait. User sees words as they speak and can re-record immediately. Lambda only generates credentials (~200ms), the WebSocket runs in the browser.

### Data Flow
1. `App.jsx` reads `?survey=<id>` from the URL. No param ‚Üí "Invalid survey link" screen.
2. Questions fetched by `surveyId`, sorted by `order`.
3. User completes info step (name + email), then answers questions (typed or voice).
4. On submit, a `Submission` is created with all responses tagged by `surveyId`.
5. Each survey's questions and submissions are isolated by `surveyId`.

## Data Models

### Survey
`id` ¬∑ `name` (required) ¬∑ `questions` (hasMany ‚Üí Question) ¬∑ `submissions` (hasMany ‚Üí Submission)

### Question
`id` ¬∑ `text` (required) ¬∑ `order` (integer, required) ¬∑ `conditions` (JSON, optional ‚Äî branching logic) ¬∑ `surveyId` (FK) ¬∑ `survey` (belongsTo)

### Submission
`id` ¬∑ `respondentName` (optional) ¬∑ `respondentEmail` (required) ¬∑ `responses` (JSON ‚Äî array of `{ questionId, questionText, responseText }`) ¬∑ `surveyId` (FK) ¬∑ `survey` (belongsTo)

### Authorization
- **Survey / Question:** guests read, authenticated users read + create + update.
- **Submission:** guests and authenticated users can create and read.
- **transcribeAudio mutation:** guests can invoke.
- Default auth mode: IAM (supports unauthenticated/guest access).

# Terminal 2 ‚Äî frontend

### Local Dev Flow
1. `npx ampx sandbox` ‚Üí deploys dev backend, generates `amplify_outputs.json`.
2. `npm run dev` ‚Üí Vite at `http://localhost:5173`.
3. `node seedquestions.js --name "Test"` ‚Üí prints link like `http://localhost:5173?survey=<id>`.
4. Open link to test end-to-end.
5. Backend edits in `amplify/` ‚Üí sandbox auto-redeploys. Frontend edits ‚Üí Vite hot-reloads.

### Deployment
Push to `main`. Amplify Hosting auto-builds and deploys frontend + backend.

## Environment & Secrets

No API keys needed. Everything is IAM:
- **Lambda** ‚Üí permissions via `backend.ts`.
- **Frontend** ‚Üí connects via `amplify_outputs.json`.
- **Transcribe creds** ‚Üí generated at runtime by Lambda (STS AssumeRole).
- **Local dev** ‚Üí `aws configure` with PowerUserAccess or AdministratorAccess-Amplify.

Future secrets: use Amplify's `secret()` function + `npx ampx sandbox secret set SECRET_NAME`.

## Troubleshooting

| Problem | Check |
|---|---|
| Mic permission denied | Site must be HTTPS (localhost exempt in dev) |
| WebSocket fails | Lambda creds have `transcribe:StartStreamTranscription`? Region supports Transcribe Streaming? |
| Bad transcription | Add Custom Vocabulary in Transcribe with domain terms |
| Audio gaps / cutouts | PCM sample rate must match `MediaSampleRateHertz` (both 16000) |
| High latency on mobile | Reduce audio chunk size for more frequent partials |
| iPhone mic not working | Needs HTTPS + user gesture to start audio capture in Safari |
